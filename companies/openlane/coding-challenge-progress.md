### Design Decisions (Interview Talking Points)
- **Why `as_json` instead of Jbuilder?** — Simpler for a small API. Jbuilder (a gem already in the Gemfile) uses template files for more complex JSON, but `as_json` keeps everything in the controller for a small project like this.
- **Why `includes` (eager loading)?** — Without it, Rails would run a separate SQL query for every question's user and every answer's user (the "N+1 query problem"). `includes` loads all the data in 3 queries instead of potentially dozens.
- **Why `only:` on `as_json`?** — Controls which fields are exposed. Without it, `created_at`, `updated_at`, `user_id` (the raw foreign key) would all leak into the response.

## Phase 3: Tenant API Key Authentication - COMPLETE

### What We Did

1. **Updated `Tenant` model** (`app/models/tenant.rb`):
   - Added `validates :name, presence: true` — every tenant must have a name
   - Added `validates :api_key, uniqueness: true, allow_nil: true` — API keys can't collide. `allow_nil: true` is needed because the `before_create` callback sets the key *after* validation runs on new records
   - Added `self.authenticate(api_key)` **class method** — looks up a tenant by API key:
     ```ruby
     def self.authenticate(api_key)
       return nil if api_key.blank?     # reject empty/nil keys immediately
       find_by(api_key: api_key)        # returns Tenant or nil
     end
     ```
   - Class method (`self.method`) is called on the model itself (`Tenant.authenticate(key)`), not on an instance

2. **Added `before_action :authenticate_tenant!`** as a private method in the API controller:
   ```ruby
   def authenticate_tenant!
     api_key = request.headers["X-Api-Key"]       # read the HTTP header
     @current_tenant = Tenant.authenticate(api_key)
     if @current_tenant.nil?
       head :unauthorized                          # 401 — stops the request here
     else
       @current_tenant.track_request!              # count it (Phase 4)
     end
   end
   ```
   - `before_action` runs this method *before* `index` or `show` — if it sends a 401, the action never runs
   - The request flow: HTTP request → middleware stack → routes → `before_action` (auth check) → controller action

### Files Modified
- `app/models/tenant.rb` — added validations, `authenticate`, `track_request!`
- `app/controllers/api/questions_controller.rb` — added `before_action` and `authenticate_tenant!`

### Design Decisions (Interview Talking Points)
- **Why 401 and not 403?** — 401 means "you didn't provide valid credentials" (who are you?). 403 means "you're authenticated but not allowed" (I know who you are, but no). Since a bad API key means we can't identify the caller, 401 is more semantically correct.
- **Why `head :unauthorized` instead of `render json: { error: "..." }`?** — Simpler. An empty 401 response is standard for API auth failures. The status code itself tells the caller what went wrong.
- **Why `allow_nil: true` on api_key validation?** — Gotcha we hit: Rails runs validations *before* `before_create` callbacks. So when creating a new tenant, the api_key is still nil during validation. The `before_create` callback fills it in right after validation but before the database INSERT.

### How to Test Manually

```bash
# → HTTP/1.1 401 Unauthorized

# Valid key → 200 with JSON (get a real key from the dashboard or rails runner)
curl -i -H "X-Api-Key: <paste-real-key-here>" http://localhost:3000/api/questions
# → HTTP/1.1 200 OK + JSON array of questions
```

### Key Concepts Learned
- **`before_action`** — Runs a method before every controller action. If it renders/redirects, the action is skipped. Think of it as per-controller middleware.
- **Class method vs instance method** — `self.authenticate` is called as `Tenant.authenticate(key)`. Instance methods (like `track_request!`) are called on a specific tenant object.
- **`request.headers`** — Hash-like object to access HTTP headers in Rails controllers
- **`head :unauthorized`** — Sends a 401 response with no body. Rails maps symbol names to HTTP codes (`:unauthorized` = 401, `:not_found` = 404, etc.)
- **Validation gotcha with callbacks** — Validations run before `before_create`, so you can't require a field that a callback is about to set

## Phase 4: Request Tracking - COMPLETE

### What We Did

1. **Created migration** (`db/migrate/20201014195400_add_request_count_to_tenants.rb`):
   ```ruby
   add_column :tenants, :request_count, :integer, default: 0, null: false
   ```
   - Adds a new column to the existing `tenants` table
   - `default: 0` — new tenants start at 0 without needing to set it explicitly
   - `null: false` — the database won't allow NULL values (always has a number)
   - Ran with `bin/rails db:migrate`

2. **Added `track_request!` instance method** to Tenant model:
   ```ruby
   def track_request!
     increment!(:request_count)
   end
   ```
   - `increment!` runs a single SQL query: `UPDATE tenants SET request_count = request_count + 1 WHERE id = ?`
   - This is **atomic** — even if two requests hit at the same time, the count is accurate because the database handles the addition
   - Compare to `self.request_count += 1; save!` which would be vulnerable to race conditions (read-modify-write)
   - The `!` in Ruby method names is a convention meaning "this method modifies something" (here, it saves to the database immediately)

3. **Called in `authenticate_tenant!`** — After successful auth, `@current_tenant.track_request!` bumps the count. This means only authenticated requests are counted (failed auth attempts are not).

### Files Created/Modified
- `db/migrate/20201014195400_add_request_count_to_tenants.rb` — **new file**, the migration
- `app/models/tenant.rb` — added `track_request!` method

### Design Decisions (Interview Talking Points)
- **Why `increment!` instead of `update`?** — Atomic operation. If two API requests come in simultaneously, `increment!` ensures both are counted. A read-then-write approach could lose a count.
- **Why track in `before_action` instead of middleware?** — We need to know *which* tenant is making the request, and the tenant lookup happens in the controller. Middleware runs earlier in the stack before we've identified the tenant. (The throttle middleware in Phase 7 uses the API key directly, which is a different approach.)

### Key Concepts Learned
- **`increment!`** — Atomic database update, runs a single SQL UPDATE. Safer than read-modify-write.
- **Migration for existing tables** — You don't always create new tables. `add_column` adds a column to an existing table.
- **`default:` and `null: false`** — Ensures the column always has a value, even for existing rows
- **Bang methods (`!`)** — Ruby convention: `method!` usually means it modifies the receiver or raises on failure

### What We Did

1. **Search query parameter** — Added to `Api::QuestionsController#index`:
   ```ruby
   if params[:search].present?
     questions = questions.where("title LIKE ?", "%#{params[:search]}%")
     return head :not_found if questions.empty?

   ```
   - `params[:search]` reads the `?search=term` query parameter from the URL

2. **Throttle middleware** (`lib/middleware/tenant_throttle.rb`):
   ```ruby
   class TenantThrottle
     DAILY_LIMIT = 100
     THROTTLE_PERIOD = 10  # seconds

     def initialize(app)
       @app = app
     end

     def call(env)
       # 1. Extract API key from the HTTP header
       # 2. Check daily request count in Rails.cache
       # 3. If over 100/day AND last request was < 10 seconds ago → 429
       # 4. Otherwise, increment count, record timestamp, pass request through
     end
   end
   ```
   - **Rack middleware** sits in the request pipeline *before* Rails controllers. Every request passes through it.
   - Uses `Rails.cache` (in-memory by default) to track per-key counts and timestamps
   - `expires_in: 24.hours` automatically clears old data
   - Returns HTTP 429 (Too Many Requests) when throttled
   - Registered in `config/application.rb` with `config.middleware.use TenantThrottle`

### Files Created/Modified
- `lib/middleware/tenant_throttle.rb` — **new file**, the Rack middleware

### Design Decisions (Interview Talking Points)
- **Why middleware for throttling instead of `before_action`?** — Middleware runs earlier in the request pipeline, before Rails routing and controllers. This means throttled requests are rejected faster and use fewer server resources. It's the standard approach for rate limiting.
- **Why `Rails.cache` instead of the database?** — Cache reads are much faster than database queries. Rate limiting needs to be checked on every request, so speed matters. The trade-off is that cache is ephemeral (lost on server restart), but that's fine for rate limiting — if the server restarts, rate limits resetting is acceptable.
- **Why 404 for empty search results instead of 200 with empty array?** — The challenge spec says "Return an appropriate HTML status code if no results are found." 404 signals "no resource found" which matches the intent. Some APIs prefer 200 with an empty array; either is defensible in an interview.
- **SQL injection safety** — The `?` placeholder in `where("title LIKE ?", ...)` is critical. Rails parameterizes the query, so user input can never break out of the string and execute arbitrary SQL.

### Key Concepts Learned
- **Rack middleware** — A Ruby class with `initialize(app)` and `call(env)`. `@app.call(env)` passes the request to the next middleware (or Rails). Returning `[status, headers, body]` short-circuits the pipeline.
- **Request pipeline order** — Web server → Rack middleware stack → Rails router → controller `before_action` → controller action → view rendering → response back up the stack
- **`Rails.cache`** — Key-value store. `write(key, value, expires_in:)` and `read(key)`. Defaults to `ActiveSupport::Cache::MemoryStore` in development.
- **`params`** — Hash-like object containing both URL query parameters (`?search=term`) and route parameters (`:id`). Rails merges them together.
- **HTTP 429 Too Many Requests** — Standard status code for rate limiting. Clients should wait and retry.
- **SQL injection protection** — Always use `where("column LIKE ?", value)` with the `?` placeholder, never string interpolation like `where("column LIKE '#{value}'")`
